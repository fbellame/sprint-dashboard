## Bug Report Template

When creating a new bug report, use the following structure:

1. **Summary** - Brief description of the bug
2. **Description** - Detailed explanation
3. **Steps to Reproduce** - Clear, numbered steps
4. **Expected Behavior** - What should happen
5. **Actual Behavior** - What actually happens
6. **Environment Details** - Where the bug was found
7. **Technical Investigation** - What needs to be checked
8. **Suggested Fix** - Proposed solution
9. **Acceptance Criteria** - Definition of done

## Bug Severity Levels

- **Critical**: Application crashes, data loss, security vulnerabilities
- **High**: Major functionality broken, significant user impact
- **Medium**: Minor functionality issues, workarounds available
- **Low**: Cosmetic issues, minor inconveniences

## Bug Priority Levels

- **P0**: Fix immediately (blocks release)
- **P1**: Fix in current sprint
- **P2**: Fix in next sprint
- **P3**: Fix when time permits

## Bug Status

- **Open**: Bug reported, not yet assigned
- **Assigned**: Bug assigned to developer
- **In Progress**: Developer is working on fix
- **Review**: Fix submitted, awaiting review
- **Resolved**: Fix verified and merged
- **Closed**: Bug is fixed and verified in production
